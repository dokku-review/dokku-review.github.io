dokku·review

install ci

Wire your repo to deploy a fresh app
on every pull request.

Two reusable GitHub Actions workflows live in dokku-review/actions. Call them from your app repo's workflow file — one deploys when a PR is labeled review, the other tears down when the PR closes.

// Don't have a server yet? Run the installer wizard first →

01Secrets & variables

Set these on the consumer repo (the app you want to deploy):

NameValue
REVIEW_SERVERvar The SSH target — an IP (203.0.113.10) or hostname (review.example.com). Hostnames need a separate A record pointing at the server; IPs work out of the box.
REVIEW_DOMAINvar The apps domain, e.g. review.example.com. Apps land at pr-N.<REVIEW_DOMAIN> — requires the wildcard *.<REVIEW_DOMAIN> DNS record.
REVIEW_SSH_PRIVATE_KEYsecret Private half of an SSH keypair. The public half is added to the server via dokku ssh-keys:add gha.

If you didn't generate the deploy keypair during installation:

ssh-keygen -t ed25519 -f gha-key -N '' ssh you@your-server 'cat | sudo dokku ssh-keys:add gha' < gha-key.pub gh secret set REVIEW_SSH_PRIVATE_KEY < gha-key gh variable set REVIEW_SERVER --body 'review.example.com' gh variable set REVIEW_DOMAIN --body 'review.example.com'
02Workflow file

Add this as .github/workflows/review-app.yml in your app repo:

name: Review App on: pull_request: types: [labeled, synchronize, closed] jobs: deploy: if: > github.event.action != 'closed' && contains(github.event.pull_request.labels.*.name, 'review') uses: dokku-review/actions/.github/workflows/deploy.yml@v1 with: server: ${{ vars.REVIEW_SERVER }} domain: ${{ vars.REVIEW_DOMAIN }} app: pr-${{ github.event.pull_request.number }} ref: ${{ github.event.pull_request.head.sha }} with_redis: true secrets: ssh_private_key: ${{ secrets.REVIEW_SSH_PRIVATE_KEY }} teardown: if: github.event.action == 'closed' uses: dokku-review/actions/.github/workflows/teardown.yml@v1 with: server: ${{ vars.REVIEW_SERVER }} app: pr-${{ github.event.pull_request.number }} with_redis: true secrets: ssh_private_key: ${{ secrets.REVIEW_SSH_PRIVATE_KEY }}

// @v1 pins the workflow to a tagged release. Drop to @main if you want the bleeding edge.

03Rails example

Rails apps need a couple of extras — a master key for credential decryption, the right RAILS_ENV, and (typically) an APP_HOST for url helpers. Pass those via env_vars and env_secrets:

deploy: if: > github.event.action != 'closed' && contains(github.event.pull_request.labels.*.name, 'review') uses: dokku-review/actions/.github/workflows/deploy.yml@v1 with: server: ${{ vars.REVIEW_SERVER }} domain: ${{ vars.REVIEW_DOMAIN }} app: pr-${{ github.event.pull_request.number }} ref: ${{ github.event.pull_request.head.sha }} with_redis: true env_vars: | RAILS_ENV=staging APP_HOST=pr-${{ github.event.pull_request.number }}.${{ vars.REVIEW_DOMAIN }} secrets: ssh_private_key: ${{ secrets.REVIEW_SSH_PRIVATE_KEY }} env_secrets: | RAILS_MASTER_KEY=${{ secrets.REVIEW_RAILS_MASTER_KEY }}
Why a separate env_secrets

The workflow merges env_vars (plain inputs) and env_secrets (multi-line GitHub secret) before calling dokku config:set.

Keeping the master key in env_secrets means GitHub will redact it from any workflow log — interpolating it directly into env_vars would leak it on a failed step.

04Test it

  1. Open a PR

    Any PR will do. The workflow won't trigger yet — it waits for the review label.

  2. Label it review

    The deploy workflow fires. Watch it in the Actions tab. ~3–5 minutes for a typical Rails app.

  3. Hit the URL

    A bot comment with https://pr-N.<REVIEW_DOMAIN> appears on the PR. Click it.

  4. Close the PR

    Teardown workflow runs. App + database gone. GitHub deployment marked inactive.
